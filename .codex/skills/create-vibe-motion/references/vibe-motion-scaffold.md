# Vibe Motion Scaffold Integration

Use this reference when the target contains `shared/scaffold/README.md` or a `vibe-motion-app` generated by `create-vibe-motion`.

## Project Shape

The scaffold separates runtime from feature logic:

- `preview/*`: UI shell and preview controls.
- `remotion/*`: Remotion entry wrappers.
- `shared/scaffold/*`: reusable scaffold runtime.
- `shared/project/*`: active project plugin and composition registry.
- `shared/features/*`: feature-specific scene, timing, config, and components.

Keep runtime files generic. Put new animation code under `shared/features/<feature-name>/`.

## Feature Layout

Preferred structure:

```text
shared/features/<feature-name>/
  config/<feature-name>Defaults.js
  plugins/<feature-name>Plugin.js
  scenes/<FeatureName>Scene.jsx
  scenes/<feature-name>SceneBuilder.js
```

Use the existing `demoMotion` feature as the local template.

## Plugin Contract

A feature plugin should expose:

- `id`
- `defaultProps`
- `SceneComponent`
- `resolveSceneContext(pluginParams)`
- `getDurationInFrames({ fps, sceneContext, pluginParams })`
- `buildSceneProps({ frame, fps, loop, sceneContext, pluginParams })`
- `getLayout({ sceneContext, sceneProps, pluginParams })`
- `normalizeParamValue(...)` when preview controls need typed values
- `paramFields`, `controlPanelTitle`, and optional `controlPanelDescription`
- `waitForSceneLayoutReady` only when async layout settling is required

Animation logic in `buildSceneProps` must be frame-deterministic because Remotion can render frames out of order and in parallel.

## Implementation Procedure

1. Copy the local `demoMotion` pattern rather than changing scaffold runtime.
2. Rename identifiers, defaults, field labels, and scene builder functions to the new effect.
3. Keep `videoWidth` and `videoHeight` as layout params; keep animation params separate.
4. Build all time-varying props in the scene builder from `frame`, `fps`, `loop`, and resolved context.
5. Make the scene component mostly presentational. It should render from props, not own timeline state.
6. Register the plugin in `shared/project/projectRegistry.js`.
7. Add or update a composition entry with the plugin id.
8. Point `ACTIVE_COMPOSITION_ID` in `shared/project/projectConfig.js` to the intended composition when the user wants it active.
9. Run `pnpm run verify` from the app directory when dependencies are already installed.

## Dependency Policy

- Reuse existing dependencies first.
- If an effect needs `three`, `@react-three/fiber`, `gsap`, `lottie-web`, `@rive-app/*`, `d3`, or another library that is not installed, ask for or use package-manager approval according to the environment.
- For simple effects, prefer plain React/SVG/CSS before adding new dependencies.
- For complex 3D or particle work, adding Three.js is appropriate when it materially improves the result.

## Render Safety

- Avoid `Date.now()`, unseeded `Math.random()`, global mutable animation state, `requestAnimationFrame`, and live DOM timeline mutation for final rendered frames.
- Use stable seeded random generation for particles or repeated marks.
- Keep transparent-background support unless the user asks for a background.
- Use scalable layout derived from `videoWidth` / `videoHeight`.
