5---

KubeJS 作为魔改模组，可以修改事件逻辑、修改配方和标签，也可以添加物品、方块、流体。

你会用 JavaScript 脚本语言来编写这些内容。由于 KubeJS 支持不少 JS 语言特性，如果你对这门语言完全不了解，建议在上手前先做一些基本了解。有关特性会在之后的视频里做更具体的解释。

6---

这里推荐使用 VS Code 进行 KubeJS 开发。详细的下载方式你可以自行搜索，或者看简介和评论区里附上的文档链接。

真正需要注意的不是下载，而是打开工作区的方式。你需要把整个实例文件夹当作 VS Code 的根目录打开。

比如 Prism 或 PCL2 里的实例，通常要打开的是包含 kubejs、mods、logs、config 这些文件夹的那一层，也就是 Minecraft 实例根目录。

原因一会儿讲。现在我们先打开 kubejs 文件夹。

接下来我们先看文件结构。第一次打开 KubeJS 文件夹时，里面一般会有一个 README.txt。

它会告诉你每个文件夹大概负责什么。

这一章先抓住三个最重要的目录：server scripts、client scripts、startup scripts。

---

聚焦 server_scripts 文件夹，旁边可以用字幕标出“逻辑服务端 / 配方 / 大部分事件”。

8--- need regenerate

server scripts 是你最常接触的部分。这里的脚本运行在逻辑服务端上。

注意，这里的服务端，不是说一定要有一台远程服务器。单人世界里同样存在逻辑服务端，所以 server scripts 在单人世界里也会运行。

配方修改、大部分事件处理，以及很多和世界逻辑相关的玩法，通常都会先从这里开始。

---

切到命令或文档里的 reload 说明，画面可以保留 `/kubejs reload server_scripts`。

9--- need regenerate

server scripts 还有一个很重要的优势，就是可以热重载。

热重载的意思是，游戏已经启动以后，你改完脚本，不一定每次都要重开游戏。只改 server scripts 时，可以优先重新加载这部分脚本，让它重新进入游戏正在运行的逻辑里。

这样写脚本、看日志、再回来修改，整个反馈会快很多。

需要注意的是，如果你修改了配方和标签等数据包内容，还需要使用原版的 `/reload`。

---

聚焦 client_scripts 文件夹，然后右边的阴影展示工具提示。

10--- regenerate

client scripts 对应的是客户端。这里处理的是本地表现，比如界面、提示、渲染，或者后面进阶会接触到的按键和客户端事件。

KubeJS 的开放性比较高，所以当你深入了解客户端各种机制和函数，就将能实现一些复杂效果。

---

聚焦 startup_scripts 文件夹，画面可以切到紫黑色方块，新物品，正在使用新物品。

11---

startup scripts 更特殊。它属于游戏启动阶段，通常会放注册和初始化相关的内容。

比如物品、方块这些注册物的注册与修改，还有一些更进阶的功能，都会出现在这一类脚本里。

---

画面强调“不能热重载 / 需要重启 / 报错会阻断启动”，可以切到日志或崩溃画面。

12---

但这里也最容易让新人踩坑。startup scripts 在多数情况下，不能像 server scripts 那样热重载。

也就是说，游戏已经启动以后，你再怎么刷新，后续写下的新代码都不会在游戏里看到效果，除非你重新启动游戏。

更麻烦的是，启动端脚本一旦写错，很可能直接就给你崩掉游戏。不过，虽然今天不会讲如何让启动端脚本热重载，但是会告诉你怎么在探索学习阶段，尽量不让启动端代码把游戏崩掉。

接下来，为了避免你在之后的开发中举步维艰，我会先把最重要的类型补全与调试教给你。

---

VS Code 扩展界面，搜索 ProbeJS，展示 ProbeJS Legacy 或对应版本；展示类型文件，然后展示正常补全，并展示根目录不对时没有补全的情况

14--- regenerate

先说 ProbeJS。它不是 KubeJS 的本体，但对于写 KubeJS 来说非常重要。

ProbeJS 做的事情，是把当前实例里的一部分运行时信息，比如你的整合包里有哪些物品、方块和事件，生成给 VS Code 类型系统看的文件。

这样你写脚本时，VS Code 才能知道有哪些事件、对象和方法可以用。

这也是刚才为什么要把整个实例当作工作区打开。ProbeJS 生成的类型文件依靠其生成的 jsconfig 实现实际效果。只打开 kubejs 文件夹，补全一定会失效。

---

画面展示进入世界后执行 `/probejs dump`，然后回到 VS Code，出现类型文件或补全。

15--- regenerate

第一次使用时，先确认实例里已经安装了 KubeJS 和 ProbeJS。对于当前 1.20.1 的这套文档，默认参考的是 ProbeJS Legacy。

然后进入任意一个世界，执行 `/probejs dump`。

这个命令会根据当前已经加载好的模组环境，导出一份 VS Code 能识别的类型文件。模组列表变了，版本换了，或者补全明显不对了，就应该重新 dump 一次。

---

VS Code 里输入 `ItemEvents.` 或 `event.player.`，展示补全；鼠标 Cmd/Ctrl 点击跳转类型。

16---

有了 ProbeJS 以后，写脚本就不需要完全靠猜。

比如你输入 ItemEvents，编辑器会提示有哪些事件。你拿到 event.player 以后，也可以先看这个对象下面有哪些字段和方法。

它最大的价值不是少打几个字，而是让你先看清楚当前对象是什么，再决定怎么写后面的逻辑。

接下来我们写一个最小测试。文件放在 kubejs 里面的 server scripts 目录，名字可以叫 apple.js。

这个示例只做一件事：当玩家吃掉苹果时，在玩家当前位置生成一次爆炸。

选择 server scripts，是因为它反馈最快，也不需要一开始就接触客户端渲染或启动阶段注册。

---

代码画面，逐行高亮：`ItemEvents.foodEaten`、判断苹果、取得 player、createExplosion。

19---

代码本身先不用背。你只需要看懂它分成几步。

第一步，监听食物被吃下的事件。第二步，判断吃掉的东西是不是苹果。第三步，拿到触发事件的玩家。最后，在玩家所在的位置创建一次爆炸。

以后大部分 KubeJS 脚本，都可以先按这个思路拆开看：事件什么时候触发，事件里有什么对象，拿到对象以后要调用什么方法。

---

进入游戏，执行 `/kubejs reload server_scripts`，吃苹果，触发爆炸。

20---

保存脚本以后，回到游戏里执行 `/kubejs reload server_scripts`。

这里要注意，重载只是让新脚本被重新读取，它本身不会触发事件。

所以执行完命令以后，你还需要真的吃下一颗苹果。只有这个动作发生了，ItemEvents.foodEaten 才会被触发。

如果没有效果，画面切回 VS Code 和日志窗口，展示检查列表。

如果没有效果，先不要急着乱改。

最重要的是，不要把重载脚本和触发事件混淆。命令不会替你吃苹果。

接下来是调试。起步阶段最有用的工具，就是 console 输出和日志文件。

你可以先在脚本最外层写一行 console info，用来确认文件有没有被加载。再在事件里面写一行，用来确认事件有没有真的触发。

如果示例放在 server scripts，优先看 logs 里的 kubejs server log。

---

画面展示三层日志：script loaded、event fired、event.player。

23---

调试时要分层确认。

第一层，文件有没有被加载。第二层，事件有没有触发。第三层，关键对象有没有拿到。

如果文件加载了，但事件没有触发，就不要继续怀疑 player 或 explosion。问题大概率在触发条件上。

如果事件触发了，但后面逻辑失败，再去检查对象、方法和具体报错。

---

画面切回 startup_scripts，强调 try catch 和“实验阶段不要让启动脚本直接崩游戏”。

24--- regenerate

这套方法对 startup scripts 尤其重要。

启动端脚本一旦报错就会直接崩掉游戏。所以在实验阶段，不确定的代码不要一股脑塞进去。

先用最小逻辑测试，再用日志确认执行到了哪里。必要时，把有风险的一小段代码放进 try catch 里。
